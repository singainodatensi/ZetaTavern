# ZetaTavern

ZetaTavern は、**ブラウザ完結**で動く AI ストーリーテリングアプリです。  
SillyTavern 的なキャラクターライブラリと、Zeta 的なストーリー進行管理を組み合わせ、Gemini 系モデルを直接利用できる構成を目指しています。

- リポジトリ: https://github.com/singainodatensi/ZetaTavern
- 公開 URL: https://singainodatensi.github.io/ZetaTavern/

---

## これは何のアプリか

ZetaTavern は、単なるチャット UI ではありません。  
現在は大きく次の 5 層で構成されています。

1. **ストーリー管理**  
   複数ストーリーを持ち、それぞれに独立した設定・履歴・記憶を持てる

2. **キャラクターライブラリ**  
   キャラごとの口調、説明、アバター、タグを再利用できる

3. **ストーリーテリング UI**  
   小説表示 / チャット表示、吹き出し、編集、モバイル最適化

4. **ロアと記憶**  
   ワールドロア、セッションロア、関係メモ、キャラ状態、候補提案

5. **ローカル保存 + 任意の Dropbox 同期**  
   IndexedDB を主保存先にしつつ、必要ならクラウドへ退避できる

### 最近変わった重要仕様

古いドキュメントとの差分として重要です。

- 旧来の `sceneState` 編集 UI は、現行では中心機能ではありません
- `main / support / background / absent` の 4 段階出席管理 UI は廃止方向です

- 新規ストーリー作成は `window.prompt()` ではなく、アプリ内モーダルで行います

## 目指している体験

基本の利用イメージは次の通りです。

1. ユーザーがストーリーを作る
2. 主人公設定、世界設定、ルールを書く
3. ライブラリにキャラクターを登録する
4. AI がストーリーテラーとして場面を描写する
5. ユーザーが主人公として返答する
6. ロアと記憶が積み上がっていく

目標 UI は、

- プレーンなチャットボット
- ただの小説出力

ではなく、

- ビジュアルノベル
- チャット RPG
- ロールプレイ用のセッションノート

の中間に近いものです。

## コア概念

### ストーリー

ストーリーは、このアプリの中心単位です。

現在よく使うフィールドは以下です。

- `storyId`
- `title`
- `franchise`
- `franchiseContext`
- `imageBaseUrl`
- `imageDefaultOutfit`
- `storytellerPrompt`
- `worldPrompt`
- `tags[]`
- `directorSettings`
- `protagonist`
- `characters[]`
- `messages[]`
- `characterMemory`
- `relationshipMemory`
- `session_lore`
- `lore_candidates`

### キャラクターライブラリとストーリー内キャラの違い

キャラクターはグローバルにライブラリへ保存されますが、  
各ストーリーで実際に参照するのは **そのストーリーに関係するキャラだけ** です。

現在の基本ルール:

- ストーリータグがない: すべての登録キャラが対象
- ストーリータグがある: カテゴリ / タグ一致キャラだけが対象

これは、

- プロンプト量の抑制
- クロスオーバー制御
- UI ノイズ削減

のために重要です。

### ワールドロアとセッションロア

**ワールドロア** は、作品世界で安定している設定向けです。

- 地名
- 組織名
- 陣営
- 用語
- 世界のルール

**セッションロア** は、そのストーリー進行の記録向けです。

- 何が起きたか
- 今どこまで進んだか
- 主なイベント
- 関係の変化
- 今回のセッション固有の事情

### 指向入力

ユーザーは `@:` 形式で特定キャラの発言を入力できます。

例:

```txt
@:四葉「おはよう！」
@:主人公「今日は早いね」
```

これにより、主人公だけでなく会話演出そのものをある程度ユーザーが制御できます。

### 画像トークン

ストーリーに `imageBaseUrl` が設定されている場合、AI は次のような制御行を出せます。

```txt
@img:四葉|初期衣装|表情微笑
四葉:「おはよう！」
```

このとき描画側は、たとえば次のような URL を組み立てます。

```txt
https://example-host/四葉/初期衣装/表情微笑.avif
```

互換として次も受け取れます。

- Markdown 画像
- 生の `<img src="...">`

ただし、今後の本命は `@img:` 形式です。

## AI 出力の前提ルール

ZetaTavern のチャット表示は、AI の出力をそのまま流しているわけではありません。  
`js/ui.js` 側で、地の文・台詞・動作を分解して描画しています。

### 推奨出力形式

- 地の文: `【...】`
- 心理寄りの描写 / 本文: `**...**`
- 台詞: `名前:「台詞」`
- 画像制御: `@img:...`

例:

```txt
【放課後の教室。窓から夕日が差し込んでいる。】
**空気は静かだが、どこか落ち着かない。**
@img:四葉|初期衣装|表情微笑
四葉:「おはよう！」
風太郎（少し顔を上げる）
```

### これが重要な理由

この形式から外れすぎると、

- 吹き出し化できない
- 誰の発言か分からない
- 画像制御が効かない
- 編集単位が崩れる

といった問題が起きます。

## ロア機能の現在地

ロアまわりは、このアプリの中でも特に重要な層です。

### 現在の実務フロー

1. ユーザーまたは AI がロア候補を見つける
2. 候補として提示される、または手動でロアを作る
3. 必要なら検索ベースで詳細化する
4. ワールドロアまたはセッションロアへ保存する

### 現在の方針

会話からの自動ロア登録は、以前かなり緩くしていたため、

- 一般名詞が大量に入る
- 重要度の低い語ばかり増える
- モデルコストに見合わない

という問題がありました。

そのため現在は、

- 無差別な自動登録は抑制
- 候補提示と確認フローを重視
- 検索で下書きを作る方に比重を置く

という方向になっています。

## Dropbox 同期

Dropbox 同期はオプション機能です。  
主保存先はあくまで IndexedDB です。

### 重要な注意

- ローカルファーストの設計です
- データ量が増えると Dropbox 同期は重くなります
- 頻繁すぎる同期は失敗や競合の原因になります

### 関連ファイル

- [DROPBOX_SETUP.md](./DROPBOX_SETUP.md)
- `js/dropbox.js`
- `js/app.js` 内の Push / Pull / Auto Sync ロジック

### リスクになりやすい点

- 長大なストーリー履歴
- 画像アセットの増加
- 同期頻度が高すぎる設定
- Dropbox 側のロック / エラー状態

## ローカル開発

### ローカルで起動する

`file://` ではなく HTTP サーバを使ってください。

例:

```bash
cd ZetaTavern
python -m http.server 8000
```

アクセス先:

```txt
http://127.0.0.1:8000/
```

### キャッシュ周りの注意

このプロジェクトでは、

- `index.html` のクエリ版数
- `sw.js` のキャッシュ名

の両方でフロント更新を制御しています。  
表示が変わらないときは、両方を確認してください。

## 他の AI / エージェント向けガイド

このリポジトリに別の AI が入るときは、まずここを読んでください。

### 先に読むべきもの

1. この README
2. 実際のコード
3. 必要なら関連 md

古い md ファイルにある仕様は、すでに現在のコードと食い違っていることがあります。

### 重要な編集ポイント

- 見た目 / 描画を変える  
  `js/ui.js`, `style.css`

- AI の出力ルールやプロンプトを変える  
  `js/ai-client.js`

- ストーリー設定や作成フローを変える  
  `js/app.js`, `index.html`

- ロアの挙動を変える  
  `js/ai-client.js`, `js/ui.js`, `js/db.js`

- キャラスコープを変える  
  `js/story-characters.js`

- Dropbox 挙動を変える  
  `js/app.js`, `js/dropbox.js`

### よくある落とし穴

- Service Worker キャッシュのせいで修正が見えない
- `window.prompt()` のようなブラウザ依存 UI はアプリ内ブラウザで不安定
- ロア抽出を緩めすぎると一般名詞スパムになる
- Dropbox は性能と失敗率のバランスが難しい
- 旧仕様の `sceneState` や Attendance 4分類を前提にするとズレやすい
