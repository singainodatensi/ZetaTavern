# ZetaTavern

ZetaTavern は、**ブラウザ完結**で動く AI ストーリーテリングアプリです。  
SillyTavern 的なキャラクターライブラリと、Zeta 的なストーリー進行管理を組み合わせ、Gemini 系モデルを直接利用できる構成を目指しています。

- リポジトリ: https://github.com/singainodatensi/ZetaTavern
- 公開 URL: https://singainodatensi.github.io/ZetaTavern/

---

この README は、次の 3 者に向けた「現在の地図」です。

1. 初めてこのアプリを触るユーザー

## これは何のアプリか

ZetaTavern は、単なるチャット UI ではありません。  
現在は大きく次の 5 層で構成されています。

1. **ストーリー管理**  
   複数ストーリーを持ち、それぞれに独立した設定・履歴・記憶を持てる

2. **キャラクターライブラリ**  
   キャラごとの口調、説明、アバター、タグを再利用できる

3. **ストーリーテリング UI**  
   小説表示 / チャット表示、吹き出し、編集、モバイル最適化

4. **ロアと記憶**  
   ワールドロア、セッションロア、関係メモ、キャラ状態、候補提案

5. **ローカル保存 + 任意の Dropbox 同期**  
   IndexedDB を主保存先にしつつ、必要ならクラウドへ退避できる

## 目指している体験

基本の利用イメージは次の通りです。

1. ユーザーがストーリーを作る
2. 主人公設定、世界設定、ルールを書く
3. ライブラリにキャラクターを登録する
4. AI がストーリーテラーとして場面を描写する
5. ユーザーが主人公として返答する
6. ロアと記憶が積み上がっていく

目標 UI は、

- プレーンなチャットボット
- ただの小説出力

ではなく、

- ビジュアルノベル
- チャット RPG
- ロールプレイ用のセッションノート

の中間に近いものです。

#### キャラクターコード貼り付けテンプレート

キャラクターライブラリには、**JSON ファイル取り込み**だけでなく**コード貼り付け**でも登録できます。  
ブラウザ版 Gemini / Claude / Meta AI などにキャラクター設定を作らせるときは、次の形式で出力させるとそのまま貼り付けやすいです。

対応しているのは主に次です。

- `zetatavern-character`
- `chara_card_v2`
- `chara_card_v3`
- 旧式 V1 形式
- `entries` 配列つきの複数キャラ JSON
- キャラ配列の直貼り

まずは **ZetaTavern 専用形式** を使うのが一番安定します。

**単体キャラの推奨テンプレート**

```json
{
  "spec": "zetatavern-character",
  "version": 1,
  "name": "中野四葉",
  "category": "五等分の花嫁",
  "tags": ["五等分の花嫁", "ヒロイン", "姉妹"],
  "description": "中野家の四女。明るく運動神経が高く、感情表現が素直。",
  "personality": "人懐っこく前向き。頼まれると断れない。自分より他人を優先しやすい。",
  "mes_example": "四葉:「おはようー！」\n四葉:「えへへ、今日は何するの？」",
  "avatarBase64": ""
}
```

**複数キャラまとめ取り込みテンプレート**

```json
{
  "spec": "zetatavern-characters",
  "version": 1,
  "entries": [
    {
      "name": "中野一花",
      "category": "五等分の花嫁",
      "tags": ["五等分の花嫁", "姉妹"],
      "description": "中野家の長女。大人びた雰囲気を持つ女優志望。",
      "personality": "余裕があり、からかい上手。時々本音を隠す。",
      "mes_example": "一花:「お姉さんに任せなさい？」",
      "avatarBase64": ""
    },
    {
      "name": "中野四葉",
      "category": "五等分の花嫁",
      "tags": ["五等分の花嫁", "姉妹"],
      "description": "中野家の四女。明るく元気で、身体能力が高い。",
      "personality": "天真爛漫で協力的。思い込みで突っ走ることもある。",
      "mes_example": "四葉:「やったー！」",
      "avatarBase64": ""
    }
  ]
}
```

**他の AI に渡す依頼文テンプレート**

```txt
以下のキャラクターを、ZetaTavern に取り込める JSON 形式で出力してください。

条件:
- JSONコードブロックのみを出力する
- 形式は zetatavern-character を使う
- 項目は name / category / tags / description / personality / mes_example / avatarBase64
- avatarBase64 は画像がない場合は空文字でよい
- description は外見・立場・背景の要約
- personality は性格・価値観・行動傾向
- mes_example は口調が分かる短い会話例を2〜4行
- 日本語で書く

対象キャラクター:
中野四葉
作品:
五等分の花嫁
```

**補足**

- `category` は作品名や所属カテゴリとして扱われる
- `tags` はストーリータグ一致の判定にも使える
- `avatarBase64` が空でも取り込み可能
- コードブロックのまま貼り付けてよい
- 既定のカテゴリ / タグを UI 側で上書きして登録することもできる

## コア概念

### ストーリー

ストーリーは、このアプリの中心単位です。

現在よく使うフィールドは以下です。

- `storyId`
- `title`
- `franchise`
- `franchiseContext`
- `imageBaseUrl`
- `imageDefaultOutfit`
- `storytellerPrompt`
- `worldPrompt`
- `tags[]`
- `protagonist`
- `characters[]`
- `messages[]`
- `characterMemory`
- `relationshipMemory`
- `session_lore`
- `lore_candidates`

ストーリーの雰囲気、ジャンル、展開速度、恋愛濃度などは、個別スライダーではなく `worldPrompt` と `storytellerPrompt` に文章で書きます。これにより、AIは世界観設定を中心にテイストを判断します。

`tags[]` に `テスト` または `test` が含まれるストーリーは、動作確認用のテストモードとして扱われます。この場合、AI はストーリーテラーではなく通常の対話型AIとして振る舞い、シナリオ本文、キャラクター台詞、選択肢、セッションロア更新、ストーリープラン更新を原則行いません。エラー報告や検索テストをしやすくするための例外モードです。

### キャラクターライブラリとストーリー内キャラの違い

キャラクターはグローバルにライブラリへ保存されますが、  
各ストーリーで実際に参照するのは **そのストーリーに関係するキャラだけ** です。

現在の基本ルール:

- ストーリータグがない: すべての登録キャラが対象
- ストーリータグがある: カテゴリ / タグ一致キャラだけが対象

これは、

- プロンプト量の抑制
- クロスオーバー制御
- UI ノイズ削減

のために重要です。

### ワールドロアとセッションロア

**ワールドロア** は、作品世界で安定している設定向けです。

- 地名
- 組織名
- 陣営
- 用語
- 世界のルール

**セッションロア** は、そのストーリー進行の記録向けです。

- 何が起きたか
- 今どこまで進んだか
- 主なイベント
- 関係の変化
- 今回のセッション固有の事情

### 指向入力

ユーザーは `@:` 形式で特定キャラの発言を入力できます。

例:

```txt
@:四葉「おはよう！」
@:主人公「今日は早いね」
```

これにより、主人公だけでなく会話演出そのものをある程度ユーザーが制御できます。

### 画像トークン

ストーリーに `imageBaseUrl` が設定されている場合、AI は次のような制御行を出せます。

```txt
@img:四葉|初期衣装|表情微笑
四葉:「おはよう！」
```

このとき描画側は、たとえば次のような URL を組み立てます。

```txt
https://example-host/四葉/初期衣装/表情微笑.avif
```

互換として次も受け取れます。

- Markdown 画像
- 生の `<img src="...">`

ただし、今後の本命は `@img:` 形式です。

## AI 出力の前提ルール

ZetaTavern のチャット表示は、AI の出力をそのまま流しているわけではありません。  
`js/ui.js` 側で、地の文・台詞・動作を分解して描画しています。

### 推奨出力形式

- 地の文: `【...】`
- 心理寄りの描写 / 本文: `**...**`
- 台詞: `名前:「台詞」`
- 画像制御: `@img:...`

例:

```txt
【放課後の教室。窓から夕日が差し込んでいる。】
**空気は静かだが、どこか落ち着かない。**
@img:四葉|初期衣装|表情微笑
四葉:「おはよう！」
風太郎（少し顔を上げる）
```

### これが重要な理由

この形式から外れすぎると、

- 吹き出し化できない
- 誰の発言か分からない
- 画像制御が効かない
- 編集単位が崩れる

といった問題が起きます。

## ロア機能の現在地

ロアまわりは、このアプリの中でも特に重要な層です。

### 現在の実務フロー

1. ユーザーまたは AI がロア候補を見つける
2. 候補として提示される、または手動でロアを作る
3. 必要なら検索ベースで詳細化する
4. ワールドロアまたはセッションロアへ保存する

### 現在の方針

会話からの自動ロア登録は、以前かなり緩くしていたため、

- 一般名詞が大量に入る
- 重要度の低い語ばかり増える
- モデルコストに見合わない

という問題がありました。

そのため現在は、

- 無差別な自動登録は抑制
- 候補提示と確認フローを重視
- 検索で下書きを作る方に比重を置く

という方向になっています。

## Dropbox 同期

Dropbox 同期はオプション機能です。  
主保存先はあくまで IndexedDB です。

### 基本方針

- ローカルファーストです。通常操作は IndexedDB を更新します
- Dropbox は複数端末共有とバックアップ用です
- 同期対象はストーリー、キャラクター、ワールドロア、設定、画像アセットです
- Tavily API キーなど、同期すべきでない秘匿設定は Dropbox に含めません
- 同期中はヘッダーのクラウドアイコンが同期中表示になります

### 現行の保存形式

以前は大きな JSON に全データを詰める形でしたが、現在は v2 形式で分割保存します。

```txt
/ZetaTavern/v2/
├── manifest.json
├── manifest.backup.json
├── manifest.prev.json
├── settings.json
├── lore/
│   └── {作品キー}.json
├── characters/
│   └── {characterId}__{名前}.json
└── stories/
    └── {storyId}__{タイトル}__{タグ}/
        ├── meta.json
        └── messages_0001.json
```

画像アセットは従来通り `/ZetaTavern_Assets/` 側に保存します。

### manifest の役割

`manifest.json` は同期目次です。  
各ストーリー、キャラクター、ロアファイル、アセットのパスや更新時刻を持ちます。

- `manifest.json`: 通常使用する同期目次
- `manifest.backup.json`: 予備バックアップ
- `manifest.prev.json`: 直前の目次

manifest が壊れた場合は、backup / prev の順に復旧を試みます。

### Push の仕様

自動同期では、可能な限り差分 Push を行います。

- チャット更新: 該当ストーリーの `meta.json` と必要な message chunk を更新
- ロア更新: 変更された作品ロアファイルのみ更新
- キャラクター更新: 変更されたキャラクター JSON と必要アセットを更新
- 削除: tombstone 設定を使い、クラウド側の該当項目を整理

ただし、差分同期の基準になる manifest が無い場合や、履歴が短くなった場合などはフル Push に倒れることがあります。

### Pull の仕様

起動時同期では、ローカルに保存した前回の manifest スナップショットとクラウド manifest を比較し、差分 Pull を行います。

- 変更されたストーリーだけ取得
- 変更されたキャラクターだけ取得
- 変更された作品ロアだけ取得
- 削除された項目だけローカルから削除
- 不足している画像アセットだけ取得

この比較用スナップショットが無い端末では、初回だけ全体取得になります。  
一度 Pull / Push が成功するとスナップショットが保存され、次回以降は差分判定できます。

手動 Pull は「クラウドから復元する」操作なので、原則として全体復元扱いです。

### 更新判定と timestamp

Dropbox から復元したデータを保存するときは、通常の編集保存とは別に同期専用保存を使います。  
これは、復元しただけで `timestamp` が現在時刻に書き換わり、次回 Push で全項目が更新扱いになる事故を避けるためです。

関連する実装:

- `db.saveStoryFromSync`
- `db.saveCharacterFromSync`
- `dropbox_remote_manifest_snapshot`

### 同期中表示

同期中はヘッダーのクラウドアイコンを同期中表示にします。  
タブ復帰や完了タイマーで、実処理中にアイコンだけ灰色へ戻らないよう、アプリ側で同期中状態を優先します。

### デバッグログ

差分 Pull 時は、DevTools に次のようなログを出します。

```txt
差分Pull判定: 前回manifestと比較します。
差分Pull判定: ロア 0/12 ファイルを取得対象にしました。
差分Pull判定: キャラクター 0/80 件を取得対象にしました。
差分Pull判定: ストーリー 1/20 件を取得対象にしました。
```

`x/y` が常に `y/y` になる場合、受け側ではなく、送信側の端末がフル Push して manifest 全体を更新扱いにしている可能性があります。

### 注意点

- データ量が増えると、初回同期や手動 Pull は重くなります
- 別端末で同時に編集すると、後から同期した内容が優先される場合があります
- 自動同期頻度を高くしすぎると、Dropbox API エラーやロック競合の原因になります
- 401 はアクセストークン期限切れで発生することがあり、リフレッシュしてリトライします
- Dropbox 側に一時ロックが残った場合、期限切れ後に自動解除されることがあります

- `js/state.js`  
  メモリ上の状態と購読通知

- `js/story-characters.js`  
  ストーリータグに応じたキャラスコープ判定

- `js/dropbox.js`  
  Dropbox 認証、manifest 管理、分割保存、差分 Push / Pull

## 永続化ストレージ

IndexedDB では現在、主に次のストアを使います。

- `settings`
- `assets`
- `characters`
- `stories`
- `world_lore`

画像は `assets` ストアに Blob として保存し、  
ストーリーやキャラクターからは ID 参照します。

Dropbox 同期用に、`settings` には次のような内部状態も保存します。

- `dropbox_remote_manifest_updated_at`
- `dropbox_remote_manifest_snapshot`
- `dropbox_sync_tombstones`

これらは同期差分判定や削除伝播に使います。通常ユーザーが直接編集するものではありません。

## ローカル開発

### ローカルで起動する

`file://` ではなく HTTP サーバを使ってください。

例:

```bash
cd ZetaTavern
python -m http.server 8000
```

アクセス先:

```txt
http://127.0.0.1:8000/
```

### キャッシュ周りの注意

このプロジェクトでは、

- `index.html` のクエリ版数
- `sw.js` のキャッシュ名

の両方でフロント更新を制御しています。  
表示が変わらないときは、両方を確認してください。

## 他の AI / エージェント向けガイド

このリポジトリに別の AI が入るときは、まずここを読んでください。

### 先に読むべきもの

1. この README
2. 実際のコード
3. 必要なら関連 md

古い md ファイルにある仕様は、すでに現在のコードと食い違っていることがあります。

### 重要な編集ポイント

- 見た目 / 描画を変える  
  `js/ui.js`, `style.css`

- AI の出力ルールやプロンプトを変える  
  `js/ai-client.js`

- ストーリー設定や作成フローを変える  
  `js/app.js`, `index.html`

- ロアの挙動を変える  
  `js/ai-client.js`, `js/ui.js`, `js/db.js`

- キャラスコープを変える  
  `js/story-characters.js`

- Dropbox 挙動を変える  
  `js/app.js`, `js/dropbox.js`

### よくある落とし穴

- Service Worker キャッシュのせいで修正が見えない
- `window.prompt()` のようなブラウザ依存 UI はアプリ内ブラウザで不安定
- ロア抽出を緩めすぎると一般名詞スパムになる
- Dropbox は性能と失敗率のバランスが難しい
- 旧仕様の `sceneState` や Attendance 4分類を前提にするとズレやすい
